## 🎯 一、客户端开发目标

### 1.1 功能范围

| 模块 | 功能 | 优先级 |
|------|------|--------|
| **工作台** | 用户信息、快捷入口、今日数据、待办事项 | P0 |
| **智能体列表** | 按导航标签展示、搜索、查看详情 | P0 |
| **AI 对话** | 发送消息、接收AI回复、Markdown渲染、历史记录 | P0 |
| **飞书数据** | 查看飞书卡片、查询记录、筛选 | P1 |
| **网页卡片** | 展示网页卡片、点击跳转 | P1 |
| **个人中心** | 查看资料、修改头像、修改密码 | P1 |
| **咨询日历** | 日历展示、查看咨询安排 | P2 |

### 1.2 技术要求

- ✅ uniapp + Vue 3
- ✅ 支持多端发布：iOS / Android / H5 / 小程序
- ✅ 使用 uniCloud 云函数 + 云数据库
- ✅ 权限过滤：自动按用户城市和部门过滤数据
- ✅ 响应式设计：适配不同屏幕尺寸

**Day 57-59: API 服务封装**
- [ ] 创建 API 服务封装 `utils/api-service.js`：
  ```javascript
  // utils/api-service.js
  const uniCloud = require('@dcloudio/uni-cloud')
  
  class ApiService {
    static async call(objectName, method, data = {}) {
      try {
        uni.showLoading({ title: '加载中...' })
        const cloudObject = uniCloud.importObject(objectName)
        const res = await cloudObject[method](data)
        uni.hideLoading()
        
        if (res.code === 0) {
          return res.data
        } else {
          throw new Error(res.message || '请求失败')
        }
      } catch (error) {
        uni.hideLoading()
        uni.showToast({ title: error.message, icon: 'none' })
        throw error
      }
    }
    
    // 用户相关
    static user = {
      login: (mobile, password) => ApiService.call('user', 'login', { mobile, password }),
      profile: () => ApiService.call('user', 'profile'),
      updateProfile: (data) => ApiService.call('user', 'updateProfile', data)
    }
    
    // 智能体相关
    static agent = {
      list: (params) => ApiService.call('agent', 'list', params),
      detail: (id) => ApiService.call('agent', 'detail', { id })
    }
    
    // 对话相关
    static chat = {
      send: (agentId, message, history) => 
        ApiService.call('chat', 'send', { agentId, message, history }),
      getHistory: (agentId) => ApiService.call('chat', 'getHistory', { agentId })
    }
    
    // ... 其他云对象
  }
  
  export default ApiService
  ```

### 4.3 云对象协调规则

#### 规则 1: 方法命名区分

客户端使用的方法命名要与管理端区分：
- ✅ 客户端：`agent.list()` - 简单列表
- ✅ 管理端：`agent.adminList()` - 复杂筛选列表

#### 规则 2: 权限统一使用中间件

所有查询必须使用统一的权限中间件：
```javascript
const { checkPermission } = require('common/permission')

// 在云对象方法中
const uid = this.getUniIdToken().uid
const filter = await checkPermission(uid, 'view', 'agent')
```

#### 规则 3: 变更申请流程

如需修改云对象：
1. 提出申请（在协作文档或 Issue）
2. 说明原因和影响范围
3. 双方确认后再修改
4. 修改后立即通知并联调

### 4.4 数据库协调规则

#### 如需新增字段

**示例**：客户端需要统计智能体查看次数

**申请模板**:
```markdown
## 数据库变更申请

**提出人**: 张三（客户端团队）  
**日期**: 2025-11-20  
**表名**: agents  
**变更类型**: 新增字段  

### 变更内容
- 新增字段: `view_count` (number) - 查看次数
- 默认值: 0
- 索引: 否

### 变更原因
客户端需要统计智能体的查看次数，用于热门推荐

### 影响范围
- 客户端: 需要在 `agent/detail` 方法中增加 view_count++
- 管理端: 需要在管理界面展示此字段

### 是否通过
- [ ] 客户端确认
- [ ] 管理端确认
```

## ⚠️ 五、风险与注意事项

### 5.1 风险清单

| 风险 | 应对措施 |
|------|---------|
| **云函数冲突** | 提前与管理端沟通，使用不同的方法名 |
| **权限漏洞** | 统一使用权限中间件，不要绕过权限检查 |
| **进度延期** | 及时上报，调整计划或寻求支援 |
| **真机测试问题** | 提前准备测试设备，预留时间修复 |

### 5.2 注意事项

- ⚠️ **不要直接修改云对象**，必须先与管理端确认
- ⚠️ **不要修改数据库 Schema**，必须先提出申请
- ⚠️ **不要绕过权限中间件**，否则会有安全漏洞
- ⚠️ **及时提交代码**，每周五前提交到 `dev-client` 分支
- ⚠️ **及时沟通问题**，不要等到联调时才发现
